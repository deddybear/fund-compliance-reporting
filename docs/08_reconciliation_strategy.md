# Reconciliation Strategy

## Status

Draft

---

# 1. Purpose

This document defines the reconciliation framework used to validate computation engine outputs against approved reference results.

The objective of reconciliation is to verify that:

* Computed figures are correct
* Methodology implementation is accurate
* Configuration profiles behave as expected
* Engine changes do not introduce regressions

Reconciliation serves as the primary validation mechanism for compliance calculations.

---

# 2. Objectives

The reconciliation process aims to:

1. Validate deterministic calculations
2. Detect implementation errors
3. Verify methodology configuration behavior
4. Support regression testing
5. Provide auditable validation evidence

---

# 3. Reconciliation Architecture

```text
Approved Knowledge Graph
            ↓

Computation Engine
            ↓

Computed Figures
            ↓

Reconciliation Engine
            ↓

Reference Answer Key
            ↓

Comparison Results
            ↓

Pass / Fail Decision
```

---

# 4. Reconciliation Inputs

The reconciliation process requires:

## Computed Figures

Generated by:

```text
Deterministic Computation Engine
```

Example:

```json
{
  "aggregate_non_ig_exposure": 15.0,
  "single_issuer_exposure": 7.5,
  "duration": 5.2
}
```

---

## Reference Answer Key

Provided by:

```text
firm_A_answer_key.xlsx
```

The answer key represents the expected outputs for Firm A.

---

## Methodology Profile

Example:

```text
configs/firm_a.yaml
```

The reconciliation profile must match the methodology used to generate the answer key.

---

# 5. Reconciliation Process

The reconciliation workflow consists of five stages.

```text
Load Answer Key
        ↓

Load Computed Figures
        ↓

Normalize Values
        ↓

Compare Results
        ↓

Generate Reconciliation Report
```

---

# 6. Figure Matching Strategy

Each computed figure must map to an expected answer key entry.

Example:

| Computed Figure           | Answer Key Figure         |
| ------------------------- | ------------------------- |
| aggregate_non_ig_exposure | aggregate_non_ig_exposure |
| single_issuer_exposure    | single_issuer_exposure    |
| duration                  | duration                  |
| dv01                      | dv01                      |

A missing mapping constitutes a reconciliation failure.

---

# 7. Normalization Rules

Before comparison, values must be normalized.

---

## Percentage Values

Examples:

```text
15
15.0
15.00
```

Normalized to:

```text
15.00
```

---

## Decimal Precision

Precision shall follow methodology configuration.

Example:

```yaml
reporting:
  percentage_precision: 1
```

---

## Currency Values

Currency units must match.

Example:

```text
SGD
```

versus

```text
USD
```

Results in failure.

---

# 8. Comparison Rules

The reconciliation engine compares:

```text
Expected Value
        ↓
Actual Value
```

for every figure.

---

## Exact Match

Example:

| Figure   | Expected | Actual |
| -------- | -------- | ------ |
| Duration | 5.2      | 5.2    |

Result:

```text
PASS
```

---

## Tolerance Match

Example:

| Figure   | Expected | Actual |
| -------- | -------- | ------ |
| Duration | 5.20     | 5.21   |

If within tolerance:

```text
PASS
```

---

## Failure

Example:

| Figure   | Expected | Actual |
| -------- | -------- | ------ |
| Duration | 5.20     | 5.60   |

Result:

```text
FAIL
```

---

# 9. Tolerance Policy

Tolerance values are configuration driven.

Example:

```yaml
reconciliation:
  tolerance:
    percentage: 0.01
    duration: 0.01
    dv01: 1
```

---

## Percentage Metrics

Allowed delta:

```text
±0.01%
```

---

## Duration Metrics

Allowed delta:

```text
±0.01 years
```

---

## DV01 Metrics

Allowed delta:

```text
±1
```

---

# 10. Reconciliation Report

The reconciliation engine produces a validation report.

Example:

| Figure          | Expected | Actual | Delta | Result |
| --------------- | -------- | ------ | ----- | ------ |
| Non-IG Exposure | 15.0     | 15.0   | 0.0   | PASS   |
| Duration        | 5.2      | 5.2    | 0.0   | PASS   |
| DV01            | 82450    | 82450  | 0     | PASS   |

---

## Overall Result

```text
PASS
```

when all figures pass validation.

---

## Failure Result

```text
FAIL
```

when one or more figures fail validation.

---

# 11. Reconciliation Failure Handling

Failures must be investigated before publication.

Potential causes:

* Configuration errors
* Graph classification errors
* Aggregation errors
* Calculation defects
* Incorrect answer key mappings

---

## Example

```text
Expected:
15.0%

Actual:
18.5%
```

Investigation should determine:

```text
Classification issue?
Aggregation issue?
Calculation bug?
```

---

# 12. Regression Testing

Reconciliation acts as a regression test suite.

Whenever:

* Engine logic changes
* Graph schema changes
* Methodology configuration changes

the reconciliation process should be re-executed.

---

## Regression Workflow

```text
Code Change
        ↓

Run Computation
        ↓

Run Reconciliation
        ↓

Validate Results
```

---

# 13. Audit Requirements

Every reconciliation execution generates audit events.

Examples:

| Event                    | Description           |
| ------------------------ | --------------------- |
| RECONCILIATION_STARTED   | Validation initiated  |
| RECONCILIATION_COMPLETED | Validation completed  |
| RECONCILIATION_FAILED    | Mismatch detected     |
| RECONCILIATION_PASSED    | All figures validated |

All events are written to the append-only audit log.

---

# 14. Publication Gate

A report may only proceed to narrative generation when reconciliation succeeds.

Required workflow:

```text
Computed Figures
        ↓

Reconciliation
        ↓

PASS
        ↓

Narrative Generation
```

Forbidden workflow:

```text
Computed Figures
        ↓

Reconciliation
        ↓

FAIL
        ↓

Narrative Generation
```

---

# 15. Governance Rules

Rule 1

All figures must be reconciled before publication.

---

Rule 2

Answer keys must be version controlled.

---

Rule 3

Tolerance values must be configuration driven.

---

Rule 4

Reconciliation results must be auditable.

---

Rule 5

Failed reconciliations block report publication.

---

# 16. Summary

The Reconciliation Framework provides an independent validation layer for all compliance calculations.

The framework ensures:

* Deterministic correctness
* Methodology accuracy
* Regression protection
* Auditability
* Publication safety

By comparing computed outputs against approved reference results, the platform can demonstrate that all reported figures are accurate, reproducible, and suitable for compliance reporting.
